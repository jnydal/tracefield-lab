What the walkthrough assumes
Phase 1 text assumes:

One shared canonical month per row: the same time_period entity has January crime and January ice cream (and similarly for all 12 months).
Spearman on those 12 aligned pairs from the demo CSVs gives ρ ≈ 0.97 and p < 0.001 (the CSVs are built so both series rise through summer and fall in winter).
So the doc is describing correct resolution + scalar extract on those same 12 entities, not "any completed job named Phase 1."

What your result means
Your result	Walkthrough
p ≈ 0.66	p < 0.001
Effect size ≈ 0.07	ρ ≈ 0.97
An effect size around 0.07 with a non-significant p-value means: in the data the worker actually used, the two variables are essentially unrelated (weak/no monotonic association). That is not what you get from the intended 12 aligned months; it's what you expect when pairs are misaligned, mixed, or diluted.

The analysis worker does not join on month. It only does:

                SELECT
                    l.entity_id,
                    l.value_num AS left_num,
                    ...
                    r.value_num AS right_num,
                    ...
                FROM features l
                JOIN features r ON l.entity_id = r.entity_id
                    AND r.feature_definition_id = %s
                WHERE l.feature_definition_id = %s
So each point is (total_incidents, units_sold_thousands) on the same entity_id. If that entity is not "January for both sources," the point is wrong for the demo story.

Cause 1 — Crime and ice cream on different entities (partial overlap)
STATUS: FIXED

Root cause: Both datasets use different source_record_id naming (NYC-CRIME-2023-01 vs
ICECREAM-JAN-2023). Without an explicit join key the resolver fell back to semantic
matching, which could mis-order or skip months, producing <12 shared entities.

Fixes applied:
- Demo CSVs already include a shared canonical_month column (2023-01 … 2023-12).
  Resolution walkthrough (DEMO_WALKTHROUGH_HEATCRIME.md steps 3 and 7) documents
  using joinKeys: [canonical_month] for exact cross-dataset matching.
- Resolver worker (resolution.py) now logs a WARNING when joinKeys is empty, naming
  the canonical_month pattern explicitly.
- POST /resolution/jobs API now returns a warnings[] field in the response when:
    • joinKeys is empty (semantic-only fallback warning)
    • useAllRows=true and a joinKey is not present in the file's columns
- Resolution form UI (entity-mappings-page.tsx) now:
    • Shows a checkbox column picker for joinKeys and semanticFields when the
      selected dataset has latestFileColumns available — eliminates free-text typos
    • Displays the API warnings inline as a yellow alert after job creation
    • Help text on semantic fields notes the collapse risk (all months → one entity)

Remaining operational requirement: researchers must still use canonical_month as a
join key. The system now makes this visible and hard to accidentally skip, but does
not enforce it. See Cause 3 below for the residual semantic-match risk.


Cause 2 — Duplicate feature rows (Cartesian join)
STATUS: FIXED

Root cause: No unique constraint on (entity_id, feature_definition_id, dataset_id)
in the features table. Re-running scalar extract or concurrent runs could produce
duplicate rows; the analysis query joined all of them, inflating n and distorting ρ.

Fixes applied:
- Migration 018_features_entity_def_unique.sql adds a partial unique index:
    CREATE UNIQUE INDEX features_entity_def_dataset_uniq
        ON features (entity_id, feature_definition_id, dataset_id)
        WHERE dataset_id IS NOT NULL;
  The Kotlin scalar extract worker already does delete-then-insert per
  (dataset_id, feature_definition_id) inside a transaction, so it satisfies
  the constraint for sequential runs. The index enforces it at DB level.
- Analysis worker (worker_analysis/main.py) now deduplicates rows by entity_id
  after fetching, with a WARNING log when duplicates are detected. This guards
  against NULL-dataset-id rows that the partial index does not cover.
- Analysis worker warns when n < 5 entity pairs reach the statistics function.
- Invariant check check_no_duplicate_features() added (test/invariants/checks.py);
  registered in run_all_checks(). Covers both NULL and non-NULL dataset_id.
- Integration tests added for: invariant PASS/FAIL cycle, dedup + warning under
  injected duplicate, empty-joinKeys warning.


Cause 3 — Semantic threshold creates new entities / result_summary not visible
STATUS: FIXED

Root cause: When semantic similarity falls below the threshold (default 0.85) and
createIfNoMatch=True, the resolver creates a new entity for that ice-cream month
instead of reusing the existing crime entity. The new entity has no crime feature,
so it never appears in the analysis join. The crime entity still has no ice-cream
feature. Net effect: fewer than 12 shared entities, n drops, ρ collapses.

What is already traceable: entity_map.method = 'created' marks every record that
created a new entity rather than matching an existing one. The resolution job's
result_summary column stores counts: {"exact": N, "semantic": N, "created": N,
"unmatched": N}.

What is fixed:
- With Cause 1 fixes in place (canonical_month join key + UI column picker),
  the resolver uses exact matching and semantic matching is never attempted for
  the demo scenario. This eliminates the threshold failure mode for normal use.
- Resolver logs a WARNING when joinKeys is empty (semantic-only path active).

Observability gap closed: the resolution jobs list now highlights created > 0
in amber with a "⚠ new entities created — check alignment" warning so
researchers see the signal immediately without querying entity_map directly.
